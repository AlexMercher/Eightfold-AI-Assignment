# Project Architecture

- **Input Layer**: Accepts resumes in varying formats.
- **Extraction Layer**: Extracts raw text, layout, and structured entities.
- **Canonical Model**: Standardized schema for all candidate data.
- **Normalization**: Standardizes formatting, casing, dates, etc.
- **Identity Resolution**: Determines if two records represent the same person.
- **Merge Engine**: Combines matched records.
- **Conflict Resolution**: Resolves data inconsistencies during merging.
- **Confidence Engine**: Assigns reliability scores to data points.
- **Provenance Engine**: Tracks the origin of every piece of data.
- **Projection Engine**: Projects canonical data to client-specific formats.
- **API Layer**: Exposes functionalities.

## app/pipeline/comparison/
The infrastructure responsible for effectively bucketing candidates and executing pure rule-based pair comparisons, explicitly isolating matching from similarity/merging.

### app/pipeline/comparison/comparators/
- `base.py`, `registry.py`, `engine_integration.py`: Core components of the Comparator Registry providing `BaseComparator`, thread-safe tracking, and config auto-loading.
- `utils.py`: Wraps RapidFuzz functionality to return percentage-based integer scores for string matching.
- `name.py`, `company.py`, `email.py`, `phone.py`, `github.py`, `linkedin.py`, `portfolio.py`, `location.py`, `education.py`, `job_title.py`, `skill.py`: Individual stateless, pure comparators. They retrieve candidate sub-models and yield a `ComparisonResult` object (matched, score, comparator, algorithm). Configuration dictating thresholds is loaded into the registry.

### app/pipeline/comparison/blocking/
- `base.py`, `registry.py`, `engine.py`: Core components. The `BlockingEngine` loops through active blocking strategies to map keys to candidate sets (Inverted Index). It parses O(bucket^2) subset pairs securely within 50ms for 5000+ candidates, yielding a unique pool of `CandidatePair` with provenance keys.
- `email.py`, `phone.py`, `github.py`, `linkedin.py`, `company_city.py`, `name_company.py`, `name_university.py`: Configurable, independently testable blocking strategies resolving multiple unique keys per candidate.

### app/pipeline/comparison/eligibility.py
- `EligibilityEngine`: Implements a boolean logic rule evaluator on top of the Comparator Registry. Takes two `Candidate`s and an `eligibility.yaml` structure, and resolves whether they pass any rule. Yields `EligibilityResult` with short-circuit insights.

## config/comparison/
- `comparators.yaml`: Specifies the matching thresholds (e.g., `85`) and algorithms (e.g., `rapidfuzz_ratio`, `exact`) for each individual comparator.
- `eligibility.yaml`: Specifies boolean rule combinations (e.g., shared_email OR (company AND name)) that trigger eligibility passing.
- `blocking.yaml`: Specifies which blocking strategies to register and use.

## app/pipeline/identity_resolution/
The orchestrator modules handling identity resolution. Converts candidate pairs with evidence into actionable outcomes `(MATCH, NO_MATCH, REVIEW)` purely by configured rules and structured logic without generating messy strings or over-merging.

- `models.py`: Declares immutable data objects (`EvidenceBundle`, `RuleResult`, `SimilarityResult`, `DecisionResult`) along with `Explanation` models structure.
- `evidence_collector.py`: Iterates `ComparatorRegistry` exactly once per candidate pair, emitting a stateless `EvidenceBundle`.
- `rule_engine.py`: Loads `rules.yaml` mapping conditions to `RuleResult` enforcing absolute priority.
- `similarity_engine.py`: Evaluates integer-based similarity leveraging `weights.yaml`, safely excluding completely missing fields to avoid false deflation.
- `decision_engine.py`: Employs `thresholds.yaml` acting deterministically on output from previous engines (without direct knowledge of business logic) to determine Identity Resolution Result.
- `engine.py`: Wires modules sequentially: `EvidenceCollector -> RuleEngine -> SimilarityEngine -> DecisionEngine`.

## config/identity_resolution/
- `rules.yaml`: Deterministic exact mapping scenarios (e.g., matching GitHub exactly terminates to MATCH).
- `weights.yaml`: Score distribution for non-terminated profiles.
- `thresholds.yaml`: Threshold numbers indicating matched profiles versus profiles requiring manual intervention.

## app/pipeline/resolution/
The final pipeline stage responsible for constructing unified candidate profiles from identity resolution results.

- `models.py`: Defines immutable models (`ValidatedCluster`, `ConflictResult`, `ProvenanceRecord`, `ConfidenceResult`, `MergeResult`, `ProjectedCandidate`, `ProjectionResult`).
- `coordinator.py`: `ResolutionCoordinator` orchestrates the execution flow: `graph -> validator -> merge -> confidence -> projection`.
- `graph.py`: `CandidateGraph` builds an adjacency list from match results and extracts connected components (clusters).
- `validator.py`: `ClusterValidator` inspects clusters for contradictions and annotates them with `ClusterStatus`, never mutating topology.
- `conflict_resolver.py`: `ConflictResolver` independently handles scalar conflicts using frequency, priority, and deterministic tie-breaking.
- `merge_engine.py`: `MergeEngine` reduces candidates field-by-field, invoking the conflict resolver and attaching field-level `ProvenanceRecord`s.
- `confidence_engine.py`: `ConfidenceEngine` evaluates the merged profile based on conflict counts and source density, separate from similarity scores.
- `projection_engine.py`: `ProjectionEngine` maps the final `Candidate` into the strictly typed `ProjectedCandidate` output schema.

## config/resolution/
- `conflict.yaml`: Specifies rules for conflict resolution (source priorities).
- `confidence.yaml`: Configures thresholds for HIGH/MEDIUM/LOW confidence levels.
- `projection.yaml`: Maps Canonical Candidate fields to the final output model.

## CLI & Orchestration

## PDF Hyperlink Extraction (2026-07-01)

### Root Cause Analysis
During forensic audit, it was discovered that `digital_parser.py` (using PyMuPDF) did not extract hyperlink annotations (`fitz.LINK_URI`). The text extraction only extracted visible text like "GitHub" or "LinkedIn". Later, `contact_extractor.py` used heuristic fallback regexes (`r"github[:\s/]+([a-zA-Z0-9\-_]+)"`) which would falsely match the labels and create generic placeholder URLs like `https://github` or `https://linkedin`, causing false merges in the identity resolution pipeline (e.g. Himanshu Ranjan and G. Madhusudhan).

### Files Modified
1. `app/core/digital_parser.py`: Updated `_parse_with_pymupdf` and `_extract_page_data` to extract `PDFHyperlink` annotations using `page.get_links()`.
2. `app/core/pipeline.py`: Updated to pass `hyperlinks` from `parse_result` to `field_extractor`. Also gracefully handle missing `app.security` imports.
3. `app/extraction/field_extractor.py`: Updated `extract` and `_safe_extract` signature to correctly pass specific social URLs to `contact_extractor`.
4. `app/extraction/contact_extractor.py`: Updated `extract`, `_extract_linkedin`, `_extract_github`, and `_extract_website` to accept hyperlink-verified URLs and deleted the dangerous URL inference heuristic regexes.
5. `app/core/ocr_parser.py`: Fixed `PPStructure` ImportError by gracefully catching it.
6. `app/nlp/ner_engine.py`: Handled missing Spacy model correctly by preventing `NoneType is not callable`.

### Files Created
- `app/models/schemas/pdf_hyperlink.py`: Structured schema for PDF hyperlinks.
- `tools/verification/verify_pdf_hyperlinks.py`: 7-test suite for embedded URL handling.

### Architectural Decisions
- Strict Separation of Concerns: `digital_parser` returns `PDFHyperlink` models. `field_extractor` parses domain from links and passes them explicitly as `github_url`, `linkedin_url`, `portfolio_url` down to `contact_extractor`. `contact_extractor` uses these explicit URLs over regex results.
- No URL Invention: Eliminated the flawed heuristic of inferring `https://github` from simply finding the word `GitHub`.

### Verification Results
- Test 1 (Visible URL only): PASS (Extracted visible)
- Test 2 (Embedded hyperlink only): PASS (Extracted embedded)
- Test 3 (Both visible and embedded): PASS (Embedded preferred)
- Test 4 (Neither): PASS (None extracted)
- Test 5-7 (Against Master.csv/Resumes): PASS (Himanshu Ranjan and G. Madhusudhan extracted with their correct, specific GitHub/LinkedIn URLs; generic placeholders successfully eliminated).

### Limitations
- Currently extracts social links only. Other embedded links (e.g., mailto) could be extracted if required.
- `portfolio_url` is inferred as the first non-GitHub/LinkedIn link, which could be refined if multiple random links exist.

### Execution Flow
Resume extraction now gathers raw GitHub URLs from PDF hyperlink annotations and plain text regex matches. The shared `GitHubURLClassifier` classifies each URL, filters to `PROFILE`, deduplicates by lowercase username, and chooses the most frequent username with first-seen tie breaking. If the normalized candidate still has a GitHub `SocialProfile`, `GitHubAdapter` enriches missing canonical fields (`full_name`, `headline`, `location`, `links.portfolio`, and empty `experience[0].company`) before blocking and identity resolution.

## Runtime Configurable Output Projection (2026-07-01)

## Cluster Validation Enforcement (2026-07-01)

### 1. Files Modified
- `app/pipeline/resolution/coordinator.py`
- `app/pipeline/e2e_runner.py`
- `tools/verification/verify_resolution.py`

### 2. Why the coordinator violated the architecture
The `ResolutionCoordinator` was correctly calling the `ClusterValidator` to detect contradictions, but it ignored the returned `ValidatedCluster.status`. It blindly passed all clusters, including `ClusterStatus.INVALID`, into the `MergeEngine`. This broke the architectural contract because contradictory candidate graphs (e.g., disjoint emails/phones) were forcibly merged into a single canonical identity, resulting in data corruption and logic errors downstream.

### 3. How INVALID clusters are now handled
When `ResolutionCoordinator` detects an `INVALID` cluster, it skips the multi-candidate merge process entirely. Instead, it iterates through each original candidate in the contradictory cluster, constructs an isolated `ValidatedCluster` for it (preserving the original contradictions), and passes it individually through the `MergeEngine` and `ProjectionEngine`. 

This guarantees:
- No merged fake identity is ever created for an `INVALID` cluster.
- Every standalone candidate is projected independently.
- The original contradiction graph and size metadata are preserved in the `cluster_summary.json` exactly once per cluster.
- Field-level provenance is generated deterministically by the `MergeEngine` acting on single-source inputs.
